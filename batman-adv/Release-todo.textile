h1. Release Checklist


This page contains a checklist of things that need to be done for a release.

h2. Release Scheduling

!/attachments/36/release_scheduling.svg!

* next branch is released as feature release 20_ _.a.0 after linux 3.x-rc3 is tagged
* maint branch is released on demand (for example after security issues)
* a feature release +must+ be done before 3.x-rc6. The time till 3.x-rc7 is needed to merge new features into next and submit them to the linux maintainers for 2.6.y

h2. Documentation

* [x] build changelog
* [x] update README
* [ ] write release announcement
* [x] update wiki documentation

h2. Code

* [x] make sure all code contains the release version strings
* [x] update copyright years (if needed)
* [x] remove dead code (if any)
* [x] check against kernel versions
* [x] check that maint branch is merged in next
* [x] check for open patches on the ml
* [x] git tag release


h2. Release

* [x] build package tar.gz and generate checksums
* [x] move packages/checksums to stable/sources / create symlinks from releases
* [x] update download links in the redmine downloads page
* [ ] send release mail
* [x] update IRC channel topic

h2. Distributions

* [ ] update openwrt package
* [ ] update gentoo package
* [x] update debian package

h2. Aftershow

* [x] Merge next into master branch (to get changes on README, CHANGELOG, ...)
* [x] Merge current git tag into maint (must be --ff-only or pre-tagging steps were done wrong)
* [x] Merge patches from master into next (to get all the fancy new features)
* [ ] Rebase patches in next on top of last linux -rc1 and send it to kernel maintainers for future kernels ([[SubmittingLinux]])
* [x] Enable daily check for next/master (symbol check, cppcheck, feature check)
