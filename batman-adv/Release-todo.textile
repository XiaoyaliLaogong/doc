h1. Release Checklist


This page contains a checklist of things that need to be done for a release.

h2. Release Scheduling

!/attachments/36/release_scheduling.svg!

* next branch is released as feature release 20_ _.a.0 after linux 3.x-rc3 is tagged
* maint branch is released on demand (for example after security issues)
* a feature release +must+ be done before 3.x-rc6. The time till 3.x-rc7 is needed to merge new features into next and submit them to the linux maintainers for 2.6.y


h2. Code

* [x] make sure all code contains the release version strings
* [x] update copyright years (if needed)
* [ ] remove dead code (if any)
* [ ] check against kernel versions
* [x] check that maint branch is merged in next
* [x] check for open patches on the ml
* [ ] git tag release


h2. Documentation

* [x] build changelog
* [ ] update README
* [ ] write release announcement
* [ ] update wiki documentation


h2. Release

* [ ] build package tar.gz and generate checksums
* [ ] move packages/checksums to stable/sources / create symlinks from releases
* [ ] update download links in the redmine downloads page
* [ ] send release mail
* [ ] update IRC channel topic

h2. Distributions

* [ ] update openwrt package
* [ ] update debian package


h2. Aftershow

* [ ] Merge next into master branch (to get changes on README, CHANGELOG, ...)
* [ ] Merge current git tag into maint (must be fast-forward or pre-tagging steps were done wrong)
* [ ] Merge patches from master into next (to get all the fancy new features)
* [ ] Rebase patches in next on top of linux-next and send it to kernel maintainers for future kernels ([[SubmittingLinux]])
* [ ] re-enable next check in checkstuff.sh
